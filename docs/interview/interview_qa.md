# 更新后的面试问题与标准回答

## 一、项目理解类

### 1. 这个项目一句话怎么概括？

这是一个面向开放世界游戏任务的大模型决策 Agent 系统，核心是把大模型放进一个 `Observation -> Thought -> Action -> Feedback -> Memory` 的长周期闭环中，使其能够持续感知、规划、执行和纠错。

### 2. 这个项目和普通 LLM 应用最大的区别是什么？

普通 LLM 应用通常是单轮推理，而这个项目是持续运行的决策系统。它不仅有模型推理，还有状态建模、动作执行、记忆更新、失败反馈和批量评测。

### 3. 你这个项目解决的核心痛点是什么？

主要解决大模型在开放世界任务中容易出现的三类问题：

- 长线目标遗忘
- 决策发散
- 逻辑死循环

### 4. 这个项目最核心的抽象是什么？

最核心的抽象是 Agent 决策闭环：

`Observation -> Thought -> Action -> Feedback -> Memory`

它贯穿了 Grid-world、Minecraft 和 LangChain 风格版本。

### 5. 你在这个项目里真正做了什么？

我做了环境建模、Planner 抽象、动作约束、短期记忆和失败反思设计、评测体系搭建、Minecraft 真实环境接入，以及 LangChain 风格 Planner 重构。

## 二、Agent 基础类

### 6. 为什么 Agent 需要记忆？

因为开放世界任务是长周期任务，如果没有记忆，模型很容易忘掉前面发生过什么、刚刚失败过什么、当前长期目标是什么，从而出现重复错误和逻辑断层。

### 7. 为什么你用的是动作-反馈记忆，而不是普通聊天记忆？

因为对于决策系统来说，最关键的信息不是“之前说过什么”，而是“之前做过什么、成没成功、为什么失败”。动作-反馈记忆更贴近执行逻辑，也更有利于避免重复失败。

### 8. 为什么动作空间一定要离散化？

因为真实环境只能执行有限动作。如果动作是自由文本，就容易产生模糊、不可执行或超出接口能力的输出。离散化动作空间能保证模型输出和执行层之间是一一映射的。

### 9. 为什么不能让模型一次性生成完整计划？

因为环境每一步都在变化。血量、饥饿、位置、资源、风险和可见目标都可能变化，所以一次生成到底的静态计划很快就会失效。

### 10. 开放世界 Agent 和普通工具调用型 Agent 有什么不同？

开放世界 Agent 不只是选择一个工具调用，而是在动态环境里持续做时序决策，还要处理导航、生存压力、局部可见性和动作 grounding，这比普通工具链式 Agent 更复杂。

## 三、Grid-world 设计类

### 11. 为什么先做 Grid-world，而不是直接上 Minecraft？

因为 Grid-world 是一个可控环境，更适合先验证状态建模、Planner 行为、记忆机制和评测体系。这样可以先把方法论跑通，再把复杂性迁移到真实环境。

### 12. Grid-world 里最重要的状态变量有哪些？

位置、当前 tile、相邻 tile、血量、饥饿、体力、背包、累计采集进度和目标进度。这些状态足以支撑生存、资源收集、探索和返航逻辑。

### 13. 为什么要把 `inventory` 和 `collected` 分开？

因为背包表示瞬时资源状态，而 collected 表示长期任务进度。如果只用 inventory，Agent 吃掉食物之后就会导致任务进度看起来回退，这是错误的。

### 14. 为什么要做多个场景？

因为单场景跑通不能说明 Agent 稳定。多场景可以考察不同风险密度、不同路线长度和不同资源布局下的泛化能力。

### 15. Grid-world 版本主要帮你暴露了什么问题？

主要暴露了：

- 目标完成判定逻辑
- 路径选择逻辑
- 记忆机制是否合理
- 重复失败规避是否有效
- 规则和模型决策的边界怎么划分

## 四、Planner 设计类

### 16. 为什么要保留规则版 Planner？

规则版 Planner 是一个确定性 baseline，也是调试工具。它可以帮助我区分问题到底来自环境、状态建模，还是来自模型本身。

### 17. 为什么要加 OpenAI 兼容 Planner？

因为我要验证这套系统抽象不仅能跑规则逻辑，也能接入真实大模型决策后端，这样项目就更接近真实 Agent 场景。

### 18. 为什么要再做 LangChain 风格重构？

因为当系统逻辑跑通之后，下一步重点就是模块化和可维护性。LangChain 风格重构让 Planner 层更清晰，支持结构化输出、记忆注入和更规范的决策接口。

### 19. 为什么 LangChain 版里还保留 guardrail？

因为某些规则不适合完全交给模型自由发挥，尤其是紧急生存逻辑、显式环境约束和明显的硬安全规则。混合式设计更稳。

### 20. 你对 Planner 设计的核心观点是什么？

我的观点是：大模型负责灵活高层规划，规则负责硬约束和环境 grounding，两者结合才适合真实 Agent 系统。

## 五、Minecraft 接入类

### 21. 为什么选 Mineflayer？

因为它在 Minecraft Java 生态里很成熟，支持连接服务器、读取状态、寻路、挖掘、战斗、吃东西和多种环境交互，是一个很适合做执行层的工具。

### 22. 为什么用 Python 规划、Node 执行？

因为 Mineflayer 在 Node 里最好用，而规划逻辑、实验和评测在 Python 里更方便。双进程结构能清晰解耦 Planner 和 Executor。

### 23. Minecraft 和 Grid-world 最大的不同是什么？

最大的不同是：在 Minecraft 里，“看得见目标”和“动作可执行”是两回事。真实环境里存在三维空间、障碍物、站位问题和掉落物延迟反馈。

### 24. 你在 Minecraft 里解决过最典型的问题是什么？

一个典型问题是树看得见但采不到。Bot 会报 `No reachable wood target found`。后来我把采木头逻辑改成分阶段执行，解决了目标可见但动作无法落地的问题。

### 25. 为什么 Minecraft 版本命名兼容问题重要？

因为不同版本的方块命名不一样，比如 1.12.2 是 `log / leaves`，而高版本是 `oak_log / oak_leaves`。如果不兼容，Bot 可能明明站在树边上却认为“没看到树”。

### 26. 食物获取逻辑是怎么演进的？

一开始只是抽象地“找食物”，后来逐步升级成：

- 吃现成食物
- 找成熟作物
- 找可食用动物
- 用水域作搜索锚点
- 回收掉落物
- 尽量确认库存中的可食用物确实增加

### 27. 为什么“确认捡到食物”这么重要？

因为走到目标附近不代表任务成功。真实环境里，只有当掉落物被捡起或者可食用库存增加，才能更可信地判定资源恢复成功。

## 六、评测体系类

### 28. 为什么不直接用 accuracy 做评测？

因为开放世界 Agent 不是单标签预测问题，它有多个目标。只看 accuracy 会掩盖很多实际表现差异。

### 29. `logic_coherence` 在这个项目里代表什么？

它代表 Agent 的逻辑一致性，重点看它是否频繁重复失败、是否容易陷入死循环、是否在多步推理中保持自洽。

### 30. `path_efficiency` 代表什么？

它衡量动作是不是高效。如果能完成任务但路线很绕、空转很多步，那说明 Planner 质量仍然不够好。

## 七、LangChain 相关类

### 31. 这个项目一开始就用了 LangChain 吗？

没有。一开始是手写 Planner 和闭环逻辑，先把系统本质吃透。LangChain 风格重构是在系统跑通之后做的。

### 32. 为什么不从一开始就用 LangChain？

因为如果一开始就完全依赖框架，很多底层问题不容易暴露。先自己搭闭环，更容易真正理解状态、动作、记忆和反馈是怎么协同工作的。

### 33. 你的项目和 LangChain 的概念怎么对应？

- Observation 格式化对应 Prompt 构造
- Planner 对应模型驱动决策
- 受约束动作 catalog 对应工具语义
- 动作-反馈窗口对应 Memory 注入
- 主循环对应 Agent Workflow

### 34. 结构化输出为什么重要？

因为 Planner 输出必须能被执行层稳定消费。结构化输出可以避免自由文本解析不稳定的问题。

### 35. 如果以后继续升级，LangGraph 有什么价值？

LangGraph 更适合显式状态、循环流程和多节点控制流。如果把这个项目继续做大，LangGraph 会比简单 LangChain Agent 更适合承载复杂工作流。

## 八、取舍与未来工作类

### 36. 为什么没有直接做强化学习？

因为这个项目的重点是 LLM Agent 的系统设计和长线规划能力，而不是通过 RL 去学最优策略。两者方向不同。

### 37. Minecraft 版本目前还缺什么？

还缺：

- 工具制作
- 更强的战斗能力
- 更系统的背包规划
- 更复杂的多阶段任务链
- 更稳健的导航失败恢复
- 更强的长期世界记忆

### 38. 如果继续做，你下一步会做什么？

我会补更强的生存状态机、显式任务栈、工具制作、长期记忆、可视化监控和更完整的自动实验体系。

### 39. 这个项目带给你最大的工程收获是什么？

最大的工程收获是：Agent 系统最难的不是调 API，而是把状态、动作、反馈、记忆和评测接口设计成一个稳定闭环。

### 40. 这个项目带给你最大的研究收获是什么？

最大的研究收获是：Agent 能力很大程度上取决于表示和 grounding。模型能力固然重要，但状态建模和动作语义往往更关键。

## 九、边界与诚实回答类

### 41. 这是一个完整的生产级游戏 AI 系统吗？

不是。它是一个完整的研究与工程复现项目，已经具备真实环境接入和模块化设计，但还不是生产级通用游戏 AI 平台。

### 42. 它等价于 Voyager 吗？

不等价。Voyager 是更大规模、更完整的 Minecraft Agent 框架。这个项目更偏重闭环方法论、真实接入和系统可解释性。

### 43. 你是不是已经完全解决了开放世界长线规划问题？

没有完全解决，但已经构建出一个可运行、可评测、可扩展的 Agent 系统，并且验证了核心机制在真实环境中的可迁移性。

### 44. 面试官应该从这个项目里看到你什么能力？

应该看到我不只是会调用模型，而是能设计 Agent 系统：会做状态抽象、动作约束、记忆设计、执行闭环、评测体系和真实环境工程接入。
